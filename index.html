https加密

发表于 2021-04-05 更新于 2021-06-24

未加密的风险

（1） 窃听风险：第三方可以获知通信内容。
（2） 篡改风险：第三方可以修改内容。
（3） 冒充风险：第三方可以冒充他人身份参与通信。

加密后的效果

（1） 所有信息都是加密传播，第三方无法窃听。
（2） 具有校验机制，一旦被篡改，通信双方会立刻发现。
（3） 具备身份证书，防止身份被冒充。

加密解密过程

（1）客户端： 1.支持的协议版本，比如 TLS 1.0 版 2.一个客户端生成的随机数，稍后用于生成”对话密钥” 3.支持的加密方法，比如 RSA 公钥加密 4.支持的压缩方法
（2）服务端： 1.确认协议版本 2.确认加密的算法 3.随机数 4.数字证书(证明公钥属于服务器)
（3）客户度： 1.验证证书
CA 对证书明文信息进行 hash
对 hash 后的值用 CA 的私钥加密，得到数字签名
浏览器拿到证书，得到明文 T，数字签名 S（数字签名：防止信息被篡改）
用 CA 的公钥对 S 解密，得到 S’
用证书里的 hash 算法对明文 T 进行 hash 得到 T’
对比 S’和 T’验证是否一致 2.生成一个随机数，用服务器的公钥加密这个随机数发送给服务端 3.根据三个随机数生成一个 session secret 4.用 session secret 加密”finish”信号
（4）服务端 1.服务端用私钥解密得到第三个随机数，生成 session secret 2.用 session secret 加密”finish”信号

session 的恢复

session ID

每一个对话都一个 session ID，如果对话中断，下次重连的时候，只要客户端给出这个编号，且服务器有这个编号的记录，双方就可以重新使用已有的”对话密钥”，而不必重新生成一把。session ID 是目前所有浏览器都支持的方法，但是它的缺点在于 session ID 往往只保留在一台服务器上。所以，如果客户端的请求发到另一台服务器，就无法恢复对话。

session ticket

客户端发送一个服务器在上一次对话中发送过来的 session ticket。这个 session ticket 是加密的，只有服务器才能解密，其中包括本次对话的主要信息，比如对话密钥和加密方法。当服务器收到 session ticket 以后，解密后就不必重新生成对话密钥了。目前只有 Firefox 和 Chrome 浏览器支持。

HTTP

HTTP1.0

1.每一个HTTP都要新建一个TCP连接
2.对头阻塞，前一个响应头到达后下一个请求才能发送。

HTTP1.1

1.长连接：Connection设置为keep-alive，一个TCP可以发送多个HTTP请求
2.管道化：可以不等第一个请求响应继续发送后面的请求，但响应的顺序还是按照请求的顺序返回
3.缓存处理：新增cache-control
4.端点传输

HTTP2.0

1.二进制分帧
2.多路复用
3.头部压缩
4.服务器推送

v8垃圾回收

发表于 2021-01-05 更新于 2021-06-24

内存结构分配

64 位系统下大约 1.4GB，32 位系统下 700MB。

垃圾回收机制

标记清除

  1. 垃圾收集器在运行的时候会给储存在内存中的所有变量都加上标记

  2. 去掉运行环境中的变量已经被环境中变量所引用变量的标记

  3. 依然有标记的变量就被视为准备删除的变量，原因是在运行环境中已经无法访问到这些变量了

  4. 垃圾收集器完成内存清除工作，销毁那些带标记的值并回收它们所占用的内存空间

引用计数

不常用，会有循环引用的问题

V8 垃圾回收策略

将内存分为老生代和新生代。新生代存放存活时间短的对象，老生代存放存活时间长或常驻内存的对象。对象最开始被分配到新生代，如果新生代内存空间不够，直接分配到老生代，新生代对象满足条件可以移动到老生代中。

分代内存

32 位系统新生代内存为 16MB，老生代 700MB，64 位新生代 32MB，老生代 1.4GB

新生代

新生代主要使用 Scavenge 进行管理（只复制活着的对象），主要实现是 Cheney 算法，将内存平均分为两块，使用空间叫 From，空置空间叫 To。新对象都被分配到 From 中，在空间快要占满的将活动对象复制到 To 空间，然后清空 From 的内存空间，此时，调换 From 空间和 To 空间，继续进行内存分配，当满足晋升条件时候，新生代对象会移动到老生代中。

晋升

1.如果一个对象是第二次经历从 From 空间复制到 To 空间，那么这个对象会被移动到老生代中。 2.当要从 From 空间复制一个对象到 To 空间时，如果 To 空间已经使用超过了 25%，则对象会晋升到老对象中。

老生代

老生代主要采用 Mark-Sweep（只清除死了的对象）和 Mark-Compact 算法，一个是标记清除，一个标记整理。两者不同的地方是，Mark-Sweep 在垃圾回收后会生成碎片内存，而 Mark-Compact 在清除前进行一步整理，将活动对象向一侧移动，随后清空边界的另一侧内存，这样空闲的内存是连续的，但是速度会慢一些。V8 主要使用 Mark-Sweep，在空间不足以对从新生代中晋升过来的对象进行分配时，才使用 Mark-Compact。
